# Yugami プロジェクト 実装計画（3Dボクセルグリッドとエネルギー伝播モデル）

物理演算のモデルを「面のテッセレーション」から「3Dグリッド空間における体積（Voxel）的なエネルギー伝播」へと根本的に刷新する実装計画です。

## 1. 3D空間のグリッド化（Voxel Gridの構築）
*   **ファイル**: `js/MapContext.js`, `js/Visualizer.js`
*   **実装内容**:
    *   日本近海（緯度20-50, 経度120-150）、地下0〜600kmの空間をカバーする3Dグリッドを生成します。
    *   **描画手法**: 20km間隔の場合、ノード数は約75万個となります。パフォーマンス（60FPS）とビジュアルを両立するため、WebGLの `THREE.Points`（または `THREE.InstancedMesh`）を使用し、全ノードを一度のドローコールでGPU描画するアーキテクチャに変更します。
    *   普段は薄いCyan色の「点」または「小さなワイヤーフレーム立方体」として均等に並び、空間の広がりと深度を表現します。

## 2. プレート境界のガイド化と異方性伝播（Conduit）
*   **ファイル**: `js/SeismicContext.js`, `js/Visualizer.js`（シェーダー）
*   **実装内容**:
    *   断層面そのものを「物理的な面」として描画するのをやめ、**「エネルギーが伝わりやすい方向（異方性）」のベクトル**として扱います。
    *   シェーダーの距離場計算（Distance Field）を拡張します。震源からの距離を単なる球体（Euclidean）として計算するのではなく、**断層の沈み込み方向（Dip）に沿っては減衰しにくく、垂直方向には急激に減衰する「楕円体（Ellipsoid）」の距離場**に変更します。
    *   これにより、地震が発生すると、そのエネルギーが断層面に沿って薄く広く（導管のように）拡散していく様子がボクセルグリッドの発光として可視化されます。

## 3. 震源とグリッドの結合（シェーダーベースのストレス拡散）
*   **ファイル**: `js/Visualizer.js`
*   **実装内容**:
    *   JS側では引き続き、上位50件の地震データを Uniform としてGPUに送ります。
    *   頂点シェーダー（Vertex Shader）にて、75万個のグリッドノードそれぞれが、50件のアクティブな震源から「自分（グリッド）へのストレス伝播量」を計算します。
    *   ストレスが閾値を超えたグリッドノードのサイズ（`gl_PointSize` またはスケール）と色（赤やピンクのネオン発光）を動的に変化させます。

## 4. ビジュアルの最適化と階層表現
*   **実装内容**:
    *   **ベースの空間**: 応力が0のグリッドは、透明度を極限まで下げた薄いCyanの点・枠線として描画し、日本近海の海底の「箱（シミュレーションボックス）」を表現。
    *   **エネルギーの塊**: 応力が蓄積されたグリッド群は、互いに重なり合って強烈な「発光する雲（Volume）」のようにボクセルが浮かび上がります。

## ユーザーへの確認事項
> [!IMPORTANT]
> **グリッドの描画手法（Points vs InstancedMesh）について**
> 20km間隔（約75万ノード）を個別の「立方体（Cube）」として描画すると、ローエンド環境で処理落ちが発生する可能性があります。
> 第一候補として、超高速で描画でき、ネオンのような輝きを得意とする **`THREE.Points`（パーティクル/点群）** を使用し、各グリッドを四角い光の点として表現するアプローチで進めてよろしいでしょうか？（もし「はっきりとした立方体の枠」にこだわる場合は、グリッド間隔を40km程度に広げて `InstancedMesh` を使用する代替案もあります）

> [!NOTE]
> **プレート面（旧メッシュ）の非表示化**
> ご指示の通り、これまでの「半透明な断層メッシュ面」は描画から完全に削除し、エネルギーの広がり方（光の雲の形）だけで断層の構造が浮かび上がるという、非常にストイックでプロフェッショナルな表現に切り替えます。この方針で進めて問題ないでしょうか？
